# ADR-0002: API-First Architecture

**Date:** 2026-03-15
**Status:** Accepted

---

## Context

MarketPulse must support multiple client surfaces (browser, PWA/mobile, future third-party integrations) and must allow the frontend and backend to be developed, deployed, and scaled independently. The question is whether to build a tightly-coupled monolith or an API-first architecture.

---

## Decision

Adopt an **API-first architecture** where:

1. All platform capabilities are exposed through a versioned HTTP/WebSocket API (`/api/v1/`)
2. The frontend is a separate application that consumes only the public API — it has no direct database access
3. The API is the single source of truth for business logic
4. API contracts are defined with OpenAPI (auto-generated by FastAPI)

---

## Rationale

- Enables the frontend to evolve independently of the backend
- Allows future clients (mobile app, CLI, third-party integrators) to use the same API
- Enforces a clean separation of concerns
- API documentation is always accurate because it is generated from code
- Supports team scaling: frontend and backend teams can work in parallel against a defined contract

---

## Consequences

**Positive:**
- Frontend and backend deployable independently
- Future third-party API access is already possible by design
- Clear contract boundary between services
- Easier to test each layer in isolation

**Negative / Trade-offs:**
- Initial overhead of defining API contracts before building UI
- Network round-trips vs direct function calls (mitigated by Redis caching)
- Requires CORS and auth token management between frontend and API

---

## Alternatives Considered

| Option | Reason not chosen |
|---|---|
| Next.js full-stack (API routes only) | Couples frontend and backend; limits future API consumers |
| GraphQL API | Higher implementation complexity for initial phase; REST is simpler for team onboarding |
